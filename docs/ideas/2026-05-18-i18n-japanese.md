- server: `GAME_PROMPTS` / `INTRO_PHRASES` / `CANONICAL_GUESS_PHRASES` / `INTRO_MESSAGES` / `PERSONALITY_PRESETS` を 2 言語化、`PERSONALITY_KEYS` を単一源泉に
- client: `GUESS_PATTERNS[lang]` で推測検出を言語別化、`guessPatternSync.test.ts` で INV-4 cross-language sync を CI 固定
- 単語プール: 英語既存 + 日本語 250 語（Easy 100 / Medium 100 / Hard 50）
- UI: `GAME_TEXT_BY_LANG` / `BUTTON_TEXT_BY_LANG` + `useTexts()` で全文言を言語別化、スタート画面に言語スイッチャー追加
- 論点 1（推測フォーマット）: 「言語別正規表現 + canonical phrase 強制」で決着。日本語フレーズ: 「答えは「[あなたの推測]」ですか？」

## プロダクトビジョン

Word Wrangler を **日本語ネイティブの体験**として使えるようにする。英語のお題で英語の発話を行う現状から、ユーザーの母語でゲームができる状態に拡張する。日本語を第一マイルストーンとし、将来的に他言語を追加できる構造にする。

## 解決したい課題

- 現状は UI / お題 / Gemini への指示・推測フォーマットすべてが英語前提。英語非ネイティブのユーザーには参入障壁が高い
- 日本語で遊ばせたいが、`detectWordGuess` が `Is it "X"?` パターン依存になっており、Gemini の応答言語が変わると正解判定が壊れる
- 単純な UI 翻訳ではなく、**ボットの発話言語**と**推測検出ロジック**まで言語ごとに揃える必要がある

## スコープ

3 つのレイヤを揃えて初めて「日本語対応」と言える：

1. **UI 文言の i18n**
   - ボタン・状態表示・スコア表記など
   - 言語スイッチャー（最低限: 英語 / 日本語）

2. **お題単語の言語化**
   - `client/src/data/wordWranglerWords.ts` に並ぶ 500 語は英語固定 → 言語別の単語リストを持つ
   - 単語帳サービス（B）と接続する場合、単語に `language` フィールドが必須

3. **Gemini Live の言語切替**
   - サーバ側 `server/bot.py` の `game_prompt`（"Welcome to Word Wrangler!..." 含む）を言語ごとに切替
   - 「ボットの初回挨拶 → `BotStoppedSpeaking` でゲーム開始」のフロー (`CLAUDE.md` の制約) は言語が変わっても保つ
   - パーソナリティ × 言語 の組み合わせが増える点に注意

4. **推測検出の言語対応** ⚠️ 設計論点（後述）
   - `client/src/utils/wordDetection.ts` の `TRANSCRIPT_PATTERNS.GUESS_PATTERN` は英語前提
   - 日本語では「それは X ですか？」「答えは X？」など複数表現がありえる

## 主要な論点

### 論点 1: Gemini の推測フォーマットをどう揃えるか（未確定）

3 案ある。idea 段階では未定とし、実装プラン時に確定する：

| 案 | 内容 | Pros | Cons |
|---|---|---|---|
| **構造化トークン** | サーバ prompt で `<guess>X</guess>` のような言語非依存トークンを強制 | 言語追加が正規表現 1 本で済む。検出が確実 | ボットの自然な発話を壊す可能性。Gemini が指示に従わないリスク |
| **言語別正規表現** | `Is it "X"?` / `それは X ですか` を言語ごとに持つ | 自然な発話のまま | 言語追加コスト大。表現バリエーションを網羅しきれない |
| **ハイブリッド** | 自然発話 + 末尾に隠しタグ | 自然さと検出性両立 | prompt 設計が複雑、表示時にタグ除去が必要 |

### 論点 2: 言語設定の保持先

- `localStorage`（client 側完結、最も単純）
- `ConfigurationProvider` に統合（パーソナリティと同じ層で扱う）
- `runner_args.body` 経由でサーバに渡す（**サーバ側 prompt 切替に必須**なので、いずれにせよ必要）

### 論点 3: パーソナリティとの掛け算

現状パーソナリティ別 prompt が `server/bot.py` にある。`personality × language` の組み合わせ管理が必要。
- prompt をテンプレ + 言語パーツに分離するか
- 言語ごとに完全な prompt セットを持つか

### 論点 4: お題単語の出所

- 当面は `wordWranglerWords.ts` を `wordWranglerWords.en.ts` / `wordWranglerWords.ja.ts` に分割
- 中期的には B（単語帳サービス）から API 取得に切替え
- 移行期は両方対応する必要あり

## ターゲットユーザー

- **私（作成者自身）と日本語話者の知人**。英語で遊ぶ前提だった現状から、家族・友人に気軽に試してもらえる状態へ
- 想定状況: 「日本語で遊べないと使ってもらえない」というフィードバックを既に受けている前提

## スコープ外

- 日本語以外の言語（中国語・スペイン語など）の追加。構造だけ拡張可能にしておき、実装は別 idea
- 日本語の発音認識精度チューニング（Gemini Live 側の責務）
- RTL 言語（アラビア語・ヘブライ語）対応に伴う UI レイアウト調整

## 既存システムへの影響

- `client/CLAUDE.md` の「単語推測の検出」節は論点 1 の決定に応じて要更新
- `CLAUDE.md` ルートの「`runner_args.body` のプロトコル整合性」制約に新フィールド (`language`) が追加される
- `server/bot.py` の `game_prompt` 冒頭固定挨拶は言語ごとに用意するが、「最初に一度発話を終える」フローを保つ
